Agent Sandbox Architecture

Complete design analysis of Browser Use's secure, scalable agent sandbox infrastructure — from AWS Lambda to Unikraft micro-VMs with a proxy-based control plane.

Security Infrastructure Micro-VMs · Source: @larsencc · April 2026
"

Your agent should have nothing worth stealing and nothing worth preserving.

— Larsen Cundric, Browser Use

Context

Browser Use runs millions of web agents that browse the internet and complete tasks autonomously. Their infrastructure evolved through three stages:

StageArchitectureLimitations
V1 AWS Lambda Browser-only agents; each invocation isolated No code execution capability
V2 Backend + Sandbox Tool Agent loop on backend; code execution in separate sandbox Agent coupled to REST API; redeploys kill agents; resource contention
V3 Isolated Agent + Control Plane Entire agent in Unikraft micro-VM; control plane proxies all external access Extra network hop (negligible vs LLM latency); 3 services to manage

The Two Fundamental Patterns

Pattern 1

Isolate the Tool

The agent runs on your infrastructure. Only dangerous operations (code execution, terminal) run in a separate sandbox. The agent calls the sandbox via HTTP.

Security Decoupled Scalable
Pattern 2

Isolate the Agent

The entire agent runs inside a sandbox with zero secrets. It communicates with the outside world exclusively through a control plane that holds all credentials.

Security Decoupled Scalable

Browser Use chose Pattern 2 after starting with Pattern 1. The agent becomes disposable — no secrets to steal, no state to preserve.

Overview → Detailed Design

Detailed Design Explanation

The Sandbox Layer

The same container image runs in all environments. A single config switch controls the runtime:

sandbox_mode: 'docker' | 'ukc'

Unikraft Micro-VMs Production

  • Each agent gets its own Unikraft micro-VM, booting in under 1 second
  • Provisioned via Unikraft Cloud's REST API on dedicated bare metal in AWS
  • Only 3 env variables: SESSION_TOKEN, CONTROL_PLANE_URL, SESSION_ID
  • No AWS keys, no database credentials, no API tokens
  • Scale-to-zero: Idle VMs suspend; resume instantly
  • Distributed across multiple Unikraft metros

Docker Containers Development

  • Same image, same entrypoint, same control plane protocol
  • Used on dev laptops and eval pipelines
  • Run hundreds in parallel for evaluations
  • Enables full dev/prod parity

Sandbox Hardening

Three hardening techniques execute before any agent code runs:

1

Bytecode-Only Execution

During Docker build, all Python source compiles to .pyc. Every .py file is deleted. The framework loads into memory as root — once loaded, source code no longer exists on disk.

2

Privilege Drop

Entrypoint starts as root (needed for bytecode read), then immediately drops to an unprivileged sandbox user via setuid/setgid. Everything runs unprivileged.

3

Environment Stripping

After reading tokens into Python variables, they are deleted from os.environ. If the agent inspects the environment, those variables are gone. Token is useless outside the private VPC anyway.

The Control Plane

The control plane is the sole gateway between the sandbox and the outside world — a stateless FastAPI service. Every request carries a Bearer: {session_token} header.

LLM Proxying

  • Sandbox sends only new messages
  • Control plane owns full conversation history in database
  • Reconstructs complete context per call, forwards to provider
  • Sandbox is stateless — kill and replace seamlessly
  • Enforces cost caps and handles billing

File Sync via Presigned URLs

  1. Sandbox detects changed files in /workspace
  2. Calls POST /presigned-urls with file paths
  3. Control plane generates presigned S3 URLs (session-scoped)
  4. Sandbox uploads directly to S3

The sandbox never holds an AWS credential.

The Gateway Protocol

class AgentGateway(Protocol):
    async def invoke_llm(self, new_messages, tools, tool_choice) -> LLMResponse: ...
    async def persist_messages(self, messages) -> None: ...

In production, ControlPlaneGateway sends HTTP to the control plane. In dev, DirectGateway calls the LLM directly. The agent code doesn't know which one it's using.

Comparison → ZeroBoot

ZeroBoot

Sub-millisecond VM sandboxes for AI agents via copy-on-write forking.

GitHub Rust KVM Apache-2.0 · 2k stars
⚡

0.79ms spawn latency (p50) — orders of magnitude faster than any other sandbox technology. Each fork is a real KVM virtual machine with hardware-enforced memory isolation.

How It Works

1

Template (one-time)

Firecracker boots a VM, pre-loads your runtime (Python, Node, etc.), and snapshots memory + CPU state. This takes ~15 seconds.

2

Fork (~0.8ms)

Creates a new KVM VM, maps snapshot memory as copy-on-write (mmap(MAP_PRIVATE)), and restores all CPU state. Hardware-enforced memory isolation.

3

Execute

Agent code runs inside the forked VM. Communication happens via serial I/O only — no networking inside forks.

Firecracker snapshot ──► mmap(MAP_PRIVATE) ──► KVM VM + restored CPU state
                           (copy-on-write)         (~0.8ms)

Limitations

⚠
  • CSPRNG state sharing — Forks share random state from snapshot. Kernel entropy is reseeded but userspace PRNGs (numpy, OpenSSL) need explicit reseeding.
  • Single vCPU — Each fork runs on one vCPU. Multi-vCPU is possible but not yet implemented.
  • No networking — Sandboxes communicate via serial I/O only. No outbound network.
  • Template updates — Require full re-snapshot (~15s). No incremental patching.
  • Working prototype — Not production-hardened yet.

Architectural Implications

ZeroBoot's CoW fork model is ideal for short-lived, high-throughput agent tasks where thousands of sandboxes need to spawn simultaneously (e.g., parallel evals, batch processing). The lack of networking means it's best paired with a control plane proxy pattern (like Browser Use's) or used for pure code execution where results are returned via serial I/O.

For Browser Use's architecture specifically, ZeroBoot could replace Unikraft as the sandbox layer with dramatic latency improvements (0.8ms vs <1s), but would require a serial I/O protocol instead of HTTP for sandbox-to-control-plane communication.

Emerging Architectural Patterns

Control Plane as a Tool

Expose a single tool interface to agents while encapsulating modular routing logic. Browser Use is an excellent implementation.

Sidecar Proxy Pattern

Like service meshes (Envoy/Istio), a sidecar mediates all communication — injecting auth, enforcing rate limits, preventing credential exposure.

Multi-Agent Interoperability

Google's A2A Protocol and Anthropic's MCP enabling agents from different systems to communicate via standardized protocols.

CoW Fork Sandboxing

ZeroBoot's approach — snapshot a VM, then fork it via copy-on-write for sub-millisecond spawns with hardware isolation. The next frontier.

Security Requirements Mapping

Map your security requirements to the appropriate isolation technology:

CRITICAL

Untrusted code from external users

Financial, healthcare, or government workloads with strict compliance requirements. Zero tolerance for sandbox escape.

Use: Unikraft or ZeroBoot

Hardware VM isolation (KVM). Dedicated kernel per sandbox. Unikraft adds unikernel minimalism. Both provide the strongest possible boundary short of bare metal.

HIGH

Multi-tenant platform with code execution

SaaS platforms where different customers' agents run on shared infrastructure. Need to prevent cross-tenant data access.

Use: Unikraft + Control Plane or OpenSandbox + Kata

VM-level isolation between tenants. Control plane proxy for credential separation. OpenSandbox + Kata gives you Kubernetes orchestration with VM boundaries.

MEDIUM

Internal agents with supervised access

Internal tooling, coding assistants, CI/CD agents. Agents are somewhat trusted but should be contained. Acceptable kernel-shared isolation.

Use: gVisor or OpenSandbox + gVisor

User-space kernel intercepts syscalls without full VM overhead. 10-30% I/O performance cost is acceptable. Simplest to deploy on existing Kubernetes clusters.

DEVELOPMENT

Dev/test/eval environments

Local development, CI pipelines, evaluation harnesses. Speed and density matter more than production-grade isolation.

Use: ZeroBoot (if no networking needed) or Docker (if networking needed)

ZeroBoot's 0.79ms spawn and 265KB footprint is perfect for running thousands of parallel evals. For local dev with networking, Docker with the same container image (Browser Use pattern) ensures dev/prod parity.

│  │  • Private VPC / Network Policy  • mTLS  • WAF                 │    │
│  └─────────────────────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────────────────────┘
💡

Key insight: The isolation runtime (Layer 3) is swappable. OpenSandbox makes this explicit — change the RuntimeClass and your gVisor-based sandbox becomes a Kata/Firecracker VM without modifying application code. This lets you start with gVisor for speed and upgrade to hardware VM isolation as your threat model demands it.

Security → Top Risks

Top 10 Security Risks

Prioritized by severity and likelihood.

#1

Control Plane Compromise Critical

Single point holding all credentials. Compromise = total system access to LLM keys, AWS credentials, database, and billing.

Mitigation: WAF, mTLS between services, secrets rotation, HSM for key storage, separate credential vaults per service.
#2

Insufficient Rate Limiting High

Agents can flood control plane requests causing service degradation and cost explosion.

Mitigation: Per-session rate limiting, token bucket algorithms, circuit breakers.
#3

Prompt Injection via History High

Poisoned messages manipulate LLM when conversation history is reconstructed by control plane.

Mitigation: Input sanitization, LLM output filtering, separate model instances for user vs data contexts.
#4

Session Token from Memory Medium

Sophisticated agent reads process memory to extract session token.

Mitigation: Encrypt tokens in memory, hardware-backed storage, minimize token lifetime.
#5

Presigned URL Abuse Medium

Bulk URL generation and potential data exfiltration from S3.

Mitigation: Rate limit generation, 5-minute expiry, monitor patterns.
#6

Unikraft Escape Vulnerability Medium

Less battle-tested than Firecracker; potential for undiscovered VM escape vectors.

Mitigation: Security audits, nested isolation, upstream contributions.
#7

Missing Audit Trail Medium

Actions not fully traceable — compliance failures and difficult incident response.

Mitigation: Structured logging, correlation IDs, SIEM integration.
#8

Cross-Session Data Leakage Medium

Logic bugs in session validation allowing unauthorized access to other users' data.

Mitigation: Fuzzing session logic, isolation tests, formal verification.
#9

Supply Chain Attack Medium

Compromised dependencies in sandbox image creating backdoor.

Mitigation: Image signing, SBOM generation, vulnerability scanning, minimal base images.
#10

LLM Cost Bombing Medium

Malicious agents trigger excessive LLM calls before cost cap enforcement kicks in.

Mitigation: Real-time token counting, pre-call budget checks, anomaly detection.

Recommendations → Architecture

Recommended Best Architecture

Defense-in-depth agent platform based on industry analysis and threat modeling.

Architectural Principles

Zero Trust

Never trust the agent. Never trust internal services by default. Verify explicitly on every request.

Dispose, Don't Preserve

Agents should be stateless and disposable. All valuable state lives outside the sandbox.

Proxy Everything

All external access must be mediated by an authenticated, rate-limited proxy layer.

Hardware Isolation

Use micro-VMs (Firecracker or Unikraft) — not containers — for untrusted code.

Independent Scaling

Backend, sandboxes, control plane, and external services scale on their own bottleneck.

Observable by Default

Every action, request, and decision logged with correlation IDs across all services.

Key Takeaways

1

Browser Use's architecture is production-grade. The "Isolate the Agent" pattern with a stateless control plane is the correct approach for multi-tenant platforms.

2

The control plane is the highest-value target. Hardening it with mTLS, WAF, rate limiting, and secrets rotation is the single most impactful investment.

3

Observability is the biggest gap. Structured logging, correlation IDs, and behavioral analytics are critical for a system running millions of agents.

4

Micro-VMs are the industry consensus. Firecracker, Unikraft, Kata — the industry has converged on hardware-level isolation. Containers alone are insufficient.

5

The "Isolate the Agent" pattern is winning. Browser Use, Manus, Devin, and others all run the entire agent inside isolated sandboxes.
